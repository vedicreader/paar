# Design: Consolidate `nbs/` and adopt fastcore idioms

Date: 2026-07-23
Status: Approved (design) — pending spec review

## Problem

`nbs/` has grown to 13 notebooks (+ `index.ipynb`), several of them tiny, with a
real numbering bug: `08_runtime.ipynb` and `08_tui.ipynb` share the `08` prefix.
The foundation modules are small and tightly coupled (`core` 835B, `reprs` 1.1KB,
`resolvers` 3.5KB, `providers` 2.4KB, `grid` 4.8KB, `snapshot` 4.8KB), while the
frontends are large (`fasthtml` 34KB, `tui` 29KB). Code predates a consistent
fastcore style and tests are written as standalone test functions rather than
literate example cells.

## Goals

1. Consolidate 13 notebooks → 7 (+ `index`), grouped by dependency layer.
2. Fix the `08` collision via renumbering.
3. Rewrite code with fastcore idioms (`first`, `store_attr`, `@delegates`,
   `@patch`, `L`, `dict2obj`) to shrink and clarify.
4. Convert test cells from test-function style to nbdev **literate style**:
   inline, example-driven cells interleaved with prose, with `test_eq`/asserts
   demonstrating usage right after each definition.

## Constraints (from brainstorming)

- **Entry-point module names must stay stable**: `paar.tui`, `paar.fasthtml`,
  `paar.mcp` are `[project.scripts]` targets and must keep their module names.
  Preserved via `#| default_exp` regardless of notebook filename/number.
- Other module import paths **may change** (pre-1.0, v0.0.1).
- nbdev workflow: run `nbdev-prepare` (hyphen) after every notebook. Never
  `nbdev_prepare`. hatchling build — no MANIFEST.in edits.

| `05_tui.ipynb`      | `tui`         | tui                                 | Kept module name; restructured internally |
| `06_mcp.ipynb`      | `mcp`         | mcp                                 | Kept module name |

Import paths that change (internal `from .x import` lines updated, `_modidx.py`
regenerated by nbdev):

- `paar.reprs`, `paar.resolvers`, `paar.providers` → `paar.core`
- `paar.grid` → `paar.snapshot`
- `paar.bridge`, `paar.registry` → `paar.runtime`

Ordering respects the dependency graph (agent before fasthtml, which imports
`AgentSession`).

## fastcore rewrite (behavior-preserving)

Grounded in confirmed source patterns:

- **`first`** (`fastcore.basics`) for linear "first match wins" scans:
  `resolve_for` (resolvers) and `provided_str` (providers).
- **`store_attr`** in `agent`'s `__init__` and plain (non-Textual) tui client
  classes. **Skip Textual widget subclasses** unless the swap is clean —
  Textual owns its `__init__` contract.
- **`@delegates`** on repeated kwarg chains: `roff,coff,rows,cols` and
  `offset,limit` forwarded through bridge → snapshot → grid.
- **`L`** for the list-comprehension-heavy resolver/grid children builders.
- **`dict2obj`/`AttrDict`** for `array_page`'s result dict — only if consumer
  edits in fasthtml/tui stay small; otherwise leave as plain dict.
- **`@patch`** to split the monster `fasthtml`/`tui` classes into
  one-method-per-cell literate cells (biggest readability win, idiomatic nbdev).

Also centralize the repeated `try: import numpy/pandas except: None` optional
imports where it doesn't create awkward cross-module coupling.

## Test cells → literate style

Replace blocks of standalone test functions with inline example cells:

- After each definition, a small cell that *uses* it on a realistic value and
  asserts the result with `test_eq`/`test_fail`, doubling as documentation.
- Prose markdown cells explaining intent between definition and example.
- Preserve all existing assertions' coverage — literate conversion is a
  reorganization, not a reduction. Net assertion count should not drop.

## Execution plan

- Work **one notebook at a time**, in dependency order (`00_core` first).
- Apply consolidation + fastcore + literate-test conversion in the **same pass**
  per notebook, so each notebook is touched once.
- Run `nbdev-prepare` after each notebook; require green before proceeding.
- Delete superseded old notebooks only after their content lands in the new one
  and tests pass.
- Update `index.ipynb` cross-references and `nbs/_quarto.yml` sidebar last.
- No `pyproject.toml` entry-point changes.

## Risks & mitigations

- **`@patch`-splitting the working 34KB/29KB frontends is high churn.** Mitigate
  by doing frontends last, one class at a time, with `nbdev-prepare` between
  classes; each `@patch` cell must keep exports identical (`__all__` unchanged).
- **Textual `store_attr` conflicts.** Skip where the widget base needs specific
  init ordering; only convert plain data-holder classes.
- **Renumbering breaks external `import paar.reprs` etc.** Accepted at v0.0.1.
- **Literate conversion could silently drop coverage.** Guard by comparing
  assertion counts before/after per notebook.

## Success criteria

- 7 notebooks (+ index); no numbering collisions.
- `nbdev-prepare` green end to end.
- Entry points `paar-tui` / `paar-serve` / `paar-mcp` still import and run.
- No net loss of test assertions.
- Foundation code visibly shorter via fastcore idioms; large frontends read as
  literate `@patch` cells.
